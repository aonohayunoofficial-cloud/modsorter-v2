## 0. 全体方針（確定事項）

### KPI（優先順位順）
1. **網羅性** — 中分類を妥協なく全登録する
2. **生成物の再現度** — 各中分類は実物研究に基づき実物に忠実に生成する
3. 最短性 — 上位KPIを損なわない範囲で成果を早く出す（下位）

### モード構成（トップレベル3分割）
- **手動生成**（今回の新規・選択式パラメトリック）
- **AI生成**（既存の簡易建築をリネーム。将来ローカルLLMのプロンプト変換生成のみ撤去。
  パイプラインは プロンプト→画像生成→3D化→ボクセル化→3Dプレビュー→.nbt出力 を温存）
- **クリエイト建築**（機能ブロック生成。今回ノータッチ。将来ロボット系へ）

### 進め方の原則
- 既存を壊さず横付け（追加優先）。AI生成パイプラインは資産として温存
- 出力は全モード共通で `List<GeneratedBlock>` → PreviewHtml → StructureNbtWriter に流す
- 各中分類は「実物研究 → 実装」の2段構え。配線は最短、パラメータ選定は実物基準
- LLM撤去はユーザーが「撤去」と指示したタイミングで実施（判定基準は設けない）

### 重要な技術的事実（コード確認済み）
- `GeneratedBlock` は現状 `X, Y, Z, Id` のみ。可動メタ用に nullable な `Properties`（ブロックステート辞書）追加が必要
- `StructureNbtWriter` は `Properties` 辞書を受け取り任意のブロックステートを書ける（axis/facing 対応可能）
- `StructureExpander` は決定論的展開ロジック。手動生成モードの中核資産として活用
- `volumes`（VolumePart 合成）実装済み → 双胴船に活用可能
- `Genre.cs` + `GenreCatalog.cs` + `Genres/*.json` によるデータ駆動機構が既存（10ジャンル）
- 非矩形フットプリント時は屋根が flat に強制フォールバックする制約あり（要仕様判断）

## 1. 大分類・中分類 マスターリスト（確定版 v1）

### 大分類1: プロペラ（Propeller）
中分類（翼構成で網羅）:
- 2枚羽 / 3枚羽 / 4枚羽 / 5枚以上（多翼）
- 後退翼（swept） / 直線翼 / 曲線翼
- 可変ピッチ翼 / ダクテッド（覆い付き） / 二重反転（contra-rotating）

パラメータ: 枚数・全長・根元弦長・翼端弦長・後退角・ねじり(twist/pitch)・翼形状・素材・回転角
（createmod propeller パラメータを完全内包。NACA翼型等の実物研究を含む）

### 大分類2: バルーン（Balloon）
中分類（シルエットで網羅）:
- 葉巻型（ツェッペリン式） / 涙滴型 / 球形 / 半球（熱気球式）
- 紡錘型 / 多気嚢（連結） / ブリンプ（軟式） / 硬式飛行船（リブ骨格あり）

パラメータ: 全長・幅・高さ・テーパー・シェル厚・リブ間隔・キール深さ・フィン高/長
（createmod balloon パラメータを完全内包）

### 大分類3: 船体（Hull）── 既存資産 ∪ createmod ∪ 追加要望の和集合（約30種）

既存資産（軍艦・特殊船が強み）:
- 手漕ぎボート / モーターボート / トロール船 / キャラベル / ガレオン
- 客船(liner) / 貨物船(cargo) / 駆逐艦 / 戦艦 / 空母 / 潜水艦

createmod 21種（帆船・歴史船が強み）:
- Sloop / Schooner / Frigate / Galleon* / Carrack / Longship
- Clipper / Ship of the Line / Cog / Pinnace / Dhow / Junk
- War Galley / Flying Ship / Dragon Ship / Ark / Trawler* / Tugboat
- Destroyer* / Speedboat / Yacht
  （* = 既存資産と重複、統合）

追加要望（両方に無い）:
- 双胴船(catamaran) / 潜水母艦(submarine tender)

パラメータ: 全長・全幅・喫水深・船底絞り(Bottom Pinch)・船体フレア(Hull Flare)・
フレアカーブ(Flare Curve)・タンブルホーム(Tumblehome)・シアーカーブ(Sheer)・
船首鋭さ(Bow Sharpness)・船尾オーバーハング(Stern Overhang) 等（createmod hull 準拠）
＋船種固有の上部構造（艦橋・砲塔・アイランド・司令塔・帆装）

### 大分類4: 建築物（Building）── 現実建築の外観を網羅（12種）
- 戸建て住宅（家） / 集合住宅（アパート/マンション） / 工場 / 研究所
- 倉庫 / 店舗・商業施設 / オフィスビル / 宗教建築（教会/神殿）
- 塔（タワー） / 記念建築（モニュメント） / 屋外イベント会場（ピラミッド屋根テント等）
- 公共施設（駅/学校等）

共通パラメータ: 幅・奥行き・高さ・階数・屋根形式・屋根勾配(roof_pitch)・軒の出(eave_overhang)・開口配置
用途固有パラメータ:
- 工場: 煙突本数/高さ/位置・鋸屋根(sawtooth)・大型シャッター
- 家: 煙突・ポーチ・テラス・ドーマー
- 集合住宅: 反復ユニット数・階数
- 宗教建築: ファサード様式・尖塔

屋根形式マスター（実物研究済み / Minecraft wiki 準拠）:
- 平屋根 / 片流れ(shed) / 切妻(gable) / サルトボックス / 寄棟(hip) / 半寄棟
- ダッチゲーブル / ギャンブレル / マンサード / ヘルム(塔用) / 鋸屋根(工場) / モニター屋根(工場)

規模: 大分類4 / 中分類 プロペラ10・バルーン8・船体約30・建築12

## 2. ロードマップ（フェーズ順）

### フェーズ0: 事実確定
- `ShipExpander` 船種別ビルダー末尾を読了し、船が「作り直し」か「形状パラメータ差し込み」かを確定
- 工場を建物の1バリアントとするか別Specとするかを判断
- AI生成モードの呼び出し元配線（StructureExpander→PreviewHtml→nbt）を1箇所特定

### フェーズ1: モード分岐の土台（追加のみ・既存ゼロ改変）
- `ArchitectModeHost` を3モードハブに拡張（既存 ArchitectGenClient は保持）
- UIにモード切替（手動生成／AI生成／クリエイト建築）追加
- 既存の簡易建築を「AI生成」にリネーム

### フェーズ2: 出力の器の拡張（改修）
- `GeneratedBlock` に nullable な `Properties`（ブロックステート辞書）追加
- PreviewHtml が Properties を無視しても壊れないことを確認
- axis/facing 等の可動メタ土台を確保

### フェーズ3: 手動生成モード・建築物（家）縦通し（新築＋既存流用）
- UIのスライダー/選択値 → `StructureSpec` 組み立て → `StructureExpander` に接続
- 既存パラメータ範囲（箱・屋根・開口・フットプリント・柱）で一本通す
- 「パラメータをいじると家がプレビューに出て nbt が吐ける」最初の目に見える成果
- ※着手前に「家」の実物研究（完了済み）

### フェーズ4: 建築物 中分類の網羅実装（新築＋改修）
- 各用途（工場・集合住宅・研究所・倉庫・店舗・オフィス・宗教・塔・記念・イベント会場・公共）を
  実物研究のうえ順次実装
- 不足パラメータ追加: 煙突（本数/位置/高さ）・屋根勾配・軒の出・屋根形式追加（hip/shed/sawtooth/monitor 等）・
  開口部の幅高さ可変化（大型シャッター）
- 各用途完了時に実物と突き合わせ再現度を検証

### フェーズ5: 手動生成モード・船体（パラメトリック hull）
- フェーズ0の結果に基づき、船を「船種名選択」から「断面形状パラメータ生成」へ
- createmod 準拠パラメータ（Length/Beam/Depth/Bottom Pinch/Hull Flare/Tumblehome/Sheer/Bow Sharpness/Stern Overhang 等）
- 既存 ShipExpander の決定論ロジックは再利用可能な部分を活用

### フェーズ6: 船種の網羅実装（新築）
- 船体マスター約30種を全実装（実物研究のうえ再現度担保）
- 双胴船は `volumes` 合成を活用、空母・潜水母艦・軍艦・帆船群を順次
- 未実装種はUI上でグレーアウト等で明示（マスターには全登録）

### フェーズ7: プロペラ・バルーン（新築・パラメトリック）
- 主軸に沿って断面反復する共通パラダイムで自前実装
- 各中分類（プロペラ10種・バルーン8種）を実物研究のうえ実装

### フェーズ8: AI生成のLLM撤去（改修・削除／ユーザー指示時）
- `ArchitectGenClient`・`DesignPlan`・2パスプロンプト削除
- `GenerationResult` の LLM専用フィールド（RawResponse/MatchLog）削除
- AI生成を プロンプト→画像→3D→ボクセル→プレビュー→nbt の一本道に整理

### 将来（今回スコープ外）
- クリエイト建築（機能ブロック生成）
- ロボット（二足歩行）— 可動メタ（フェーズ2の Properties）が本領を発揮

## 3. 各フェーズ共通の完了条件
- 実装後、対象の実物（写真/資料）と生成物を突き合わせ、再現度を確認する（KPI2）
- 中分類はマスターリストに全登録、未実装はUIで明示（KPI1: 網羅性を欠かさない）
- 既存モード（AI生成）が壊れていないことを確認
