# ModSorter v2 建築モード 形状語彙拡張 仕様書（確定版 v2）

## 0. この文書の位置づけ

本書は、建築モードで生成できる形状の語彙を、単一直方体から「任意平面形状（L字・コの字・凹み）」および「複数塊の合成（双胴・胴体＋部品）」へ拡張するための設計仕様である。草案段階の「要確認」7項目はリポジトリ実物（StructureExpander.cs / StructureSpec.cs / ArchitectGenClient.cs / PreviewHtml.cs / MainWindow.Architect.cs）の精査で全て解決済み。本 v2 は、確定版に対しレビューで判明した4つの補強点（決定性の順序非依存、IsRectangular の定義、開口スナップの一意規則、テスト段取り）を追記した最新版である。実装コードは含まない。ai_collab_rules.md とは独立した文書として管理する。

## 1. 背景と現状

### 1.1 既存パイプライン（確認済み）

建築モードは2段構成。第1段で自然言語の指示を Ollama(localhost:11434) の LLM に渡し、座標を持たない中間表現 StructureSpec(JSON) を生成させる（ArchitectGenClient.GenerateAsync）。第2段で StructureExpander.Expand が Spec を確定的に座標（GeneratedBlock の集合）へ展開する。揺らぎは LLM の temperature（既定0.2）と variantHint、座標展開は確定的。既存構造は「揺らぎ＋確定制御」を既に備える。

生成フローも確認済み。ArchGenerate_Click がUI欄（モデル・プロンプト・ブロックCSV・寸法W/D/H・facade）を読み、GenerateMultipleAsync で3案生成、_archCases に保持、ShowCase→RenderArchPreviewAsync でプレビュー描画、ArchExport_Click でNBT出力へ渡す。UI・プレビュー・出力は GeneratedBlock（X/Y/Z/Id）のみを扱う。

重要な既存挙動（実読で確定）: building 経路（ArchKindCombo.SelectedIndex != 1）では、UIの寸法3欄が int.TryParse を通り、2〜64の範囲検証を通過しないと生成自体が始まらない。通過した w/d/h は GenerateMultipleAsync に必ず fixed 値として渡され、spec.Width 等をLLM寸法に優先して確定する。すなわち外形サイズはUIが確定し、W/Dが未確定のままLLM寸法が生きる経路は building 経路に存在しない。彫刻モード（index==1）は別フロー（GenerateSculptureAsync）で寸法検証を通らないが、本拡張のスコープ外。

窓・ドア・アーチは ApplyOpening がセルを除去（または置換）して実現し、開口は"穴"として生成される。彫刻モードの「開口が塊で埋まる」現象は建築モードでは原理的に発生しない。

### 1.2 現状の制約（本拡張の動機）

StructureSpec は単一直方体を前提。Expand の壁生成は単一矩形外周（x==0 || x==w-1 || z==0 || z==d-1）に固定され、この判定は壁ループと土台ループの2箇所にインライン直書き。床・屋根・開口も同じ単一矩形を基準に計算される。このためL字・コの字・T字・十字など凹んだ平面形状、および双胴・胴体＋付属物など複数塊の組み合わせは生成不可能。

structure_type は building / ramp / bridge の3種のみ。ramp/bridge は専用ビルダーが通常ロジックをバイパスして自前で座標を生成する拡張ポイント。ただし形状ごとに専用ビルダーを個別追加する方式は語彙増に対し線形にコストが増え、拡張性に乏しい。

## 2. 目的とスコープ

### 2.1 目的

単一直方体の制約を撤廃し、少ない機構で多くの平面形状・複合形状を生成する。個別専用ビルダーの都度追加を避け、汎用的な合成機構へ移行する。

### 2.2 段階（フェーズ分割）

フェーズ1「マスク方式」：1つの塊の平面形状を自由化。フットプリント（平面マスク）を Spec に持たせ、壁・屋根・開口をマスクの縁に対して計算。L字・コの字・T字・十字・任意の凹み形を単一機構で表現。

フェーズ2「複数ボリューム合成」：フェーズ1で自由化した塊を複数配置・結合。双胴・胴体＋付属部品を箱（各ボリューム）のリストで表現。

両フェーズは入れ子（フェーズ2はフェーズ1の成果物を部品として利用）で、フェーズ1→フェーズ2の順を必須とする。

### 2.3 スコープ外

彫刻モード側の改修。翻訳・MOD整理等の他機能。フェーズ2の詳細設計（フェーズ1実装結果を踏まえ別途）。フェーズ1における gable/pyramid/dome の非矩形対応、ApplyOpening の周回インデックス方式、colonnade/temple/pilaster のマスク対応（いずれも次段送り）。

## 3. フェーズ1 設計方針：マスク方式

### 3.1 基本概念

平面(X-Z)上の「どのセルがフットプリントに含まれるか」を表すマスクを導入。Height(Y)方向は現行同様。壁は「マスク内セルであって、隣接4方向のいずれかがマスク外であるセル」（=マスクの縁）に立てる。床・中間床・屋根はマスク内セル全体に敷く。外周が矩形でなくても壁・屋根・開口が破綻しない。

Expand は純関数的（cells 辞書を後勝ちで構築し y→z→x でソートして返す）なので、マスクは w/d 確定直後に1回生成し、以降の全ループを「マスクに含まれるか」で条件付けるだけで導入できる。

### 3.2 マスクの表現方法（確定）

案1-B「プリセット形状＋パラメータ」（footprint_shape = rect/L/U/T/plus ＋寸法パラメータ）と、案1-C「矩形の加減算」（add リスト、sub リスト、含有 = ∪(add) − ∪(sub)）を併用。実装上はプリセットも内部で加減算リストに変換し、マスク生成経路を1系統に一本化する。外向きインターフェースは2種（プリセット／自由形状）、内部表現は加減算1種。

外形W/DはUIが固定するため、add/sub矩形はUI固定W/Dの枠内で解釈し、枠を超える座標はクランプする。

### 3.3 Spec への追加項目（確定）

footprint_shape（string?, 既定 null＝"rect"扱い）。footprint_params（形状ごとの寸法パラメータ）。footprint_add（矩形リスト）／footprint_sub（矩形リスト）。矩形型 Rect { X, Z, W, D } を新設。既定値（footprint未指定）は現行と同一の単一矩形に解決し、既存 Spec が現状と完全に同じ結果になる後方互換を必須とする。

### 3.4 Expander 改修の要点（確定）

新規ヘルパー2つ。BuildFootprint(spec, w, d) は SPEC からマスク集合を返す。IsEdge(mask, x, z) はマスク内かつ4近傍のいずれかがマスク外なら true。

一般化対象：壁の外周判定（x==0||... → IsEdge）、土台の外周判定（同）、床/中間床（マスク内全セルに限定）、flat屋根（マスク内セルのみ）、走査ループのマスク内限定。

屋根 gable/pyramid/dome/gable_stairs は矩形の中心・辺・対称性に幾何が依存するため、フェーズ1では非矩形対応しない。非矩形指定時は各分岐冒頭の1行ガードで flat へフォールバックする。

ApplyOpening は既存の ContainsKey ガードにより非矩形でも暴発しない。フェーズ1では face+offset の座標算出を「その面上でマスクの実縁セルにスナップ」する軽微補正のみを加える。矩形時は従来と同一結果になること。arch の左右セル（offset±1）が凹角でマスク外に落ちる場合も ContainsKey ガードで守られ、片側のみ抜ける見た目は許容とする。周回インデックス方式は次段送り。

pilaster は従来の x%step/z%step 判定を維持し、非矩形で見た目が乱れても可（後日精査、コメント明示）。colonnade/temple はマスク非対応（矩形専用）とし、非矩形指定時は矩形として動作させる（次段送り）。

### 3.5 決定性の担保（v2 補強）

BuildFootprint の集合演算は「add を全適用 → その後 sub を全適用」の順に固定する。add 同士・sub 同士のリスト内順序は結果に影響してはならない（マスクは集合＝HashSet で保持し、順序非依存とする）。これにより同一 Spec → 同一マスク → 同一結果が保証される。屋根等で丸めが関わる箇所は丸め方向を実装で固定する。

### 3.6 IsRectangular の定義（v2 補強）

屋根の非矩形フォールバック判定に用いる IsRectangular(mask, w, d) は「マスク要素数 == w*d」で判定する。これは厳密には「マスクが w×d の枠を完全充填しているか（=フル矩形か）」の判定であり、本仕様において「矩形」とはフル充填のみを指す。sub で内側をくり抜いた額縁状マスク等は要素数が w*d 未満のため非矩形（=flat フォールバック対象）となる。これは意図した挙動である。

### 3.7 開口スナップの一意規則（v2 補強）

L字等では同一 face（例: south）が複数の離れた辺に分割されうる。この場合、face+offset のスナップ先は「その face の辺セル群を、面に沿う座標（south/north は x 昇順、east/west は z 昇順）でソートし、offset に最も近いセルを選ぶ。同距離が並ぶときは面に沿う座標が小さい側を採用」と一意に規定する。矩形時は従来と同一結果になること。これにより非矩形でも開口位置が決定的に定まる。

### 3.8 フェーズ1の対応形状（目標）

必須：矩形（現行互換）、L字、コの字(U)、T字。可能なら：十字(plus)、任意の角削り。屋根は flat を保証し、gable/pyramid/dome は非矩形時 flat フォールバック。

## 4. フェーズ2 設計方針：複数ボリューム合成（方向のみ）

### 4.1 基本概念

Spec を「複数ボリュームのリスト」に拡張。各ボリュームはフットプリント＋高さ＋素材＋配置オフセット(X,Y,Z)を持つ独立した塊。全ボリュームを配置・重ね合わせて最終形状とする。双胴=胴体×2＋接続、胴体＋部品=大＋小群。各ボリューム内部はフェーズ1の機構を再利用する。

実装上は、Expand が純関数的に cells 辞書を返す構造を利用し、ボリュームごとに Expand 相当を呼びオフセットを加算して1つの cells へマージする合成層を1枚被せる。UI・プレビュー・NBT出力は流用可能。

### 4.2 検討が必要な論点（フェーズ1完了後）

接合部処理（重なった壁を残すか除去して貫通させるか）。素材優先順位（後勝ち＝リスト後方のボリュームが勝つ、と規約化すれば決定的）。座標系合成。LLM への説明方法。これらはフェーズ1実装結果に依存するため本書では確定しない。

## 5. 後方互換・非機能要件

後方互換：footprint 未指定・単一ボリュームの Spec は現行と完全に同一の結果を生成（既存 building/ramp/bridge を壊さない）。既存 ramp/bridge 専用ビルダーは対象外で現状維持。

決定的性：同一 Spec → 同一結果を維持。マスク集合演算は 3.5 の順序に固定。

LLM 連携：ArchitectGenClient のプロンプトに footprint の意味・選択ルール・日本語キーワード（「L字」「コの字」「双胴」等）を追記。プロンプトは既に大きいため独立ブロックを増設せず既存 structure_type/roof_type 節に寄せて最小限追記し、出力品質・トークン量を実測する。

プレビュー整合（確定・無改修）：PreviewHtml.renderBlocks はブロックを座標ごとに個別配置しフットプリント形状に非依存。中心寄せ・初期カメラ距離はバウンディングボックスから動的算出。状態付きID（[facing=...]）は split('[')[0] でベースID化済み。フェーズ1・2ともプレビュー改修は不要。

出力整合（確定・無改修）：ArchExport_Click は GeneratedBlock→StructureNbtWriter.Block（Name/X/Y/Z、Properties空）へ詰め替えるのみ。非矩形・複数ボリュームとも無改修で通る。

言語・環境：C#(.NET/WPF)。改修は StructureExpander.cs / StructureSpec.cs / ArchitectGenClient.cs の3ファイルに限定し、UI・プレビュー・NBT出力は変更しない。

## 6. 確認事項の解決状況

草案の「要確認」7項目は全て実物精査で解決済み。

1. StructureExpander 後半と共通ヘルパー：共通ヘルパーは Clamp / Pick / AxisPositions / BuildColumn / Outside / StairId の6つ。マスク・縁判定に相当する関数は無い。外周判定は壁ループと土台ループの2箇所にインライン直書き。新設は BuildFootprint と IsEdge の2つで、直書き2箇所を置換するのが起点。
2. ArchitectGenClient のパースとプロンプト：JSONモードで受け、{〜} を抜いてデシリアライズ。パース後に fixedWidth/Depth/Height でLLM寸法を上書き。プロンプトは大きいため footprint は既存節へ寄せて追記。
3. 生成フロー（MainWindow.Architect.cs、実読済み）：building 経路では UIの寸法3欄が TryParse＋2〜64範囲検証を通過しないと生成が始まらず、通過値を fixed としてLLM寸法に優先。外形はUI、形状トポロジはLLMの分担。add/sub がW/Dを超えたら BuildFootprint 内でクランプ。彫刻モード（index==1）は別フローでスコープ外。
4. 屋根の非矩形対応可否：flat はマスク化容易、gable/pyramid/dome は矩形前提の幾何に依存。フェーズ1は flat のみ対応、他は非矩形時 flat フォールバック。
5. ApplyOpening の非矩形拡張：既存 ContainsKey ガードにより安全。フェーズ1は 3.7 の縁スナップの軽微補正のみ。大改造は次段送り。
6. プレビューの非矩形・複数ボリューム描画：座標ごとの個別配置＋動的中心寄せのため無改修で対応。
7. マスク表現の最終選定：案1-B/1-C 併用、内部は加減算に一本化。NBT出力も座標集合に過ぎず無改修。

## 7. 実装順（確定）

### 7.1 フェーズ1 改修マップ（影響範囲の狭い順）

全て StructureExpander.cs / StructureSpec.cs / ArchitectGenClient.cs に閉じ、UI・プレビュー・NBT出力は無改修。

1. StructureExpander に BuildFootprint(spec, w, d)（プリセット→加減算に正規化、W/Dクランプ、add→sub の順で決定性確保、集合で保持し順序非依存）と IsEdge(mask, x, z) を追加。
2. Expand の w/d 確定直後にマスクを1回生成（既定は全 w×d＝現行同一）。
3. 壁ループ・土台ループの x==0||... 直書き2箇所を IsEdge に置換し、走査もマスク内に限定。
4. 床・中間床の全面ループをマスク内に限定。
5. BuildFlatRoof をマスク対応。
6. gable/pyramid/dome/gable_stairs の各分岐冒頭に IsRectangular（要素数==w*d）判定→非矩形なら flat フォールバックの1行ガード。
7. ApplyOpening の face+offset 算出を 3.7 の規則でマスクの実縁にスナップ（ContainsKey ガード維持）。
8. StructureSpec に footprint 系フィールド追加（既定＝現行同一）＋ Rect 型新設。
9. ArchitectGenClient プロンプトへ footprint 項目とL字・コの字の日本語キーワードを既存節と同文体で追記。
10. add/sub の W/D 超過クランプを BuildFootprint 内に実装。

第一目標（最小実装）は 1〜4＋8 で「後方互換を保ったマスク基盤＋flat屋根でL字・コの字・T字が生成可能」。次に 5〜7、最後に 9〜10。

### 7.2 テスト段取り（v2 補強）

最小実装（1〜4＋8）の段階では、9（プロンプト追記）が未実施のためLLMは footprint を出力できない。したがってこの段階の検証は、add/sub を手動指定した Spec を Expander に直接与え、Expander 単体で L字・コの字・T字が正しく展開されること、および footprint 未指定 Spec が現行と完全一致すること（後方互換）を確認する。9 完了後に、LLM 経由の end-to-end（自然言語→Spec→展開→プレビュー→ゲーム内配置）を検証する。各段階で実機確認を挟む。

### 7.3 フェーズ2 方向（フェーズ1安定後）

Expand の純関数的構造を利用し、ボリュームごとに Expand 相当を呼びオフセット加算して1つの cells にマージする合成層を被せる。素材優先順位は後勝ちで規約化し決定的性を保つ。UI・プレビュー・NBT出力は流用可能。接合部処理・座標系合成の詳細はフェーズ1実装結果を踏まえ別途確定。
